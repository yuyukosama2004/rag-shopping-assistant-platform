# 监控与告警指南

本项目使用 Prometheus、Alertmanager、Node Exporter 和 cAdvisor，覆盖应用、依赖、主机和容器四层状态。监控端口仅绑定 loopback，不应通过防火墙直接开放。

## 启动与检查

```bash
./start.sh observability-start
./start.sh observability-status
```

`start`、`install` 和成功的 `upgrade` 也会自动启动监控栈。默认保留 15 天指标，可通过 `.env` 的 `PROMETHEUS_RETENTION` 调整。

从运维电脑建立 SSH 隧道后访问界面：

```bash
ssh -L 19090:127.0.0.1:19090 -L 19093:127.0.0.1:19093 jill3@server
```

- Prometheus：`http://127.0.0.1:19090`
- Alertmanager：`http://127.0.0.1:19093`

Prometheus 的 `Status -> Targets` 中应看到四个应用、Alertmanager、Node Exporter 和 cAdvisor 全部为 `UP`。四个应用各自导出的 `biyesheji_application_health` 应为 `1`。

## 已覆盖的告警

| 告警 | 触发条件 | 级别 |
| --- | --- | --- |
| `BiyeshejiServiceUnhealthy` | 任一服务健康结果连续 1 分钟不是 `UP` | critical |
| `BiyeshejiMetricsTargetDown` | 任一核心采集目标连续 2 分钟不可达 | warning |
| `BiyeshejiHighServerErrorRate` | 单服务 5xx 比例连续 5 分钟超过 5% | warning |
| `BiyeshejiHostCpuHigh` | 主机 CPU 连续 10 分钟超过 90% | warning |
| `BiyeshejiHostDiskLow` | 根分区可用空间连续 5 分钟低于 15% | critical |
| `BiyeshejiContainerMemoryHigh` | 商城容器内存连续 10 分钟超过限制的 90% | warning |
| `BiyeshejiAiBudgetNearLimit` | 当日 AI 估算费用达到商家预算的 80% | warning |
| `BiyeshejiAiFailureRateHigh` | 10 分钟至少失败 5 次且失败率超过 20% | warning |

告警规则位于 `deploy/observability/alerts.yml`。Alertmanager 默认把告警保留在本地 UI，不包含外部账号或密钥。正式运营前应按商家使用的邮件、Slack、企业微信或其他告警渠道，在 `deploy/observability/alertmanager.yml` 中增加对应 receiver；敏感 URL 和口令应通过受限文件或 Docker secret 引用，不得提交到 Git。

## 告警演练

在维护窗口内停止一个应用容器：

```bash
docker stop biyesheji-user-service
```

约 1 分钟后，Prometheus `Alerts` 与 Alertmanager 应出现 `BiyeshejiServiceUnhealthy`。由于应用 Compose 设置了 `restart: unless-stopped`，手工 `docker stop` 不会自动重启；完成检查后执行：

```bash
docker start biyesheji-user-service
./start.sh status
```

若要验证自动恢复，应在容器内终止主进程，而不是执行 `docker stop`。完成演练后确认告警转为 resolved。

## 日志与敏感信息

四个服务默认输出 ECS JSON 日志，便于后续接入 Loki、Elasticsearch 或云日志服务。日志保留在 Docker 默认日志驱动中；不得记录 Authorization、API Key、密码、用户咨询原文或 AI 供应商响应体。需要临时人工阅读时可使用：

```bash
docker logs --since 10m biyesheji-order-service
```

生产环境应同时为 Docker 日志驱动配置轮转上限，避免日志耗尽磁盘。
