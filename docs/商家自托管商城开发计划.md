# 商家自托管 AI 商城分阶段开发计划

> 文档状态：后续开发主路线图  
> 制定日期：2026-07-13  
> 最近复查：2026-07-16
> 产品定位：单商家、单店铺、单实例、自托管的 AI 导购商城  
> 当前基线：`v0.8.0-alpha.2` 已完成阶段 0–6 的代码与内部技术验收，以及阶段 7 的自动化验收；剩余工作是公网域名 TLS、外部告警送达、真实商家试点和独立的真实支付阶段。

## 1. 产品目标

本项目不建设多商家平台，也不承担平台运营、商家入驻和平台审核职能。每位商家拥有并部署一套独立系统，自行管理店铺、商品、库存、订单、发货、售后和 AI 配置。

最终产品由两个入口组成：

1. **消费者商城**：浏览商品、搜索筛选、AI 咨询、购物车、下单、查询订单和售后状态。
2. **商家工作台**：配置店铺、管理商品与 SKU、调整库存、处理订单、录入发货信息、查看经营数据和维护 AI 知识。

### 1.1 核心成功标准

一个没有开发能力的商家，应能够按照文档完成部署，并独立完成以下闭环：

```text
初始化店主账号
  → 设置店铺信息
  → 创建商品、SKU、价格和库存
  → 上架商品
  → 消费者浏览、咨询并下单
  → 商家确认收款/处理订单
  → 商家填写物流信息并发货
  → 消费者确认完成
  → 商家备份数据并安全升级
```

### 1.2 明确不做的范围

以下能力不属于首个可用版本，除非后续重新立项：

- 多商家入驻、平台抽佣、平台管理员和商家审核。
- 同一部署承载多个店铺的多租户系统。
- 平台级资金清算、担保交易和复杂分账。
- 自建物流网络、仓储管理系统和财务 ERP。
- 多国家、多币种、多税率和跨境合规。
- 直播、拼团、分销、会员等级等大型营销体系。

## 3. 总体设计约束

### 3.1 单店单实例

- 每套数据库只服务一个商家，因此核心业务表暂不增加 `tenant_id`。
- 店铺资料使用单行配置表；系统启动时必须存在且只能存在一份有效店铺配置。
- 商家之间通过独立部署、独立数据库、独立域名和独立密钥隔离。
- 备份、升级和故障恢复以单实例为单位执行。

### 3.2 角色与权限

首版采用三个角色：

| 角色 | 权限 |
| --- | --- |
| `OWNER` | 店铺全部配置、员工管理、商品、库存、订单、AI 和系统设置 |
| `STAFF` | 商品、库存、订单处理；不可修改店主账号、系统密钥和危险设置 |
| `CUSTOMER` | 消费者商城、地址、购物车、本人订单和 AI 导购 |

约束：

- 首个 `OWNER` 通过一次性初始化命令或一次性令牌创建，禁止默认用户名和默认密码。
- 消费者注册只能获得 `CUSTOMER`，不能通过请求参数提升角色。
- `/api/merchant/**` 必须由网关鉴权，下游服务仍需执行角色和资源级校验。
- 所有商家写操作记录审计日志，包括操作者、对象、动作、结果和时间。

### 3.3 技术演进原则

- 继续使用现有 Spring Cloud 服务和 Vue 前端，首版不进行大规模重写。
- 商家工作台先集成在现有 Vue 应用的 `/merchant` 路由下，采用独立布局和路由守卫。
- 商品与 SKU 归 `product-service`；订单、库存预留和履约归 `order-service`；身份与店铺账号归 `user-service`。
- 对外消费者接口保持兼容；商家接口统一使用 `/api/merchant/**`。
- 所有数据库结构变化必须使用 Flyway，禁止在部署脚本中临时执行 `ALTER TABLE`。
- 先实现货到付款/线下收款的完整履约，再接真实支付，避免模拟支付进入生产。

## 4. 目标数据模型

以下为目标模型，具体字段通过阶段设计评审确定。

| 表/聚合 | 目的 | 关键约束 |
| --- | --- | --- |
| `t_store_setting` | 店名、Logo、联系方式、地址、营业状态、配送与售后说明 | 单实例仅一条有效配置 |
| `t_user` | 店主、店员和消费者账户 | 增加角色与账号状态；角色不可由普通资料接口修改 |
| `t_product` | SPU 级商品内容 | 草稿、上架、下架；软删除后消费者不可见 |
| `t_product_sku` | 规格组合、售价、划线价、商家编码 | SKU 编码唯一；至少一个可售 SKU 才能上架 |
| `t_product_media` | 主图、轮播图、详情媒体及排序 | 删除商品时处理媒体引用 |
| `t_stock` | SKU 总库存、锁定库存、可用库存、版本号 | `total = locked + available` 且均非负 |
| `t_stock_ledger` | 入库、扣减、释放、盘点和人工调整流水 | 每次变更具有业务来源和操作人 |
| `t_order` | 消费者订单和履约状态 | 状态转换受状态机约束 |
| `t_order_item` | 商品、SKU、价格和规格快照 | 下单后不随商品修改而改变 |
| `t_shipment` | 承运商、运单号、发货时间和备注 | 仅已确认订单可发货 |
| `t_shipping_rule` | 配送方式、基础运费、包邮门槛和自提配置 | 下单时保存规则与运费快照 |
| `t_refund_record` | 手工或渠道退款的金额、原因和处理状态 | 退款金额不得超过实付金额 |
| `t_order_operation` | 商家和系统的订单操作记录 | 只追加，不覆盖历史 |
| `t_audit_log` | 商家后台关键操作审计 | 不记录密码、Token、完整地址等敏感内容 |
| `t_ai_index_task` | 商品/知识的向量索引任务 | 可重试、可观察、可防止旧版本覆盖新版本 |

## 9. Issue 拆分与执行规则

每个阶段按以下模板拆分 GitHub Issue：

```markdown
## 背景
为什么需要这个能力，解决哪个商家或消费者问题。

## 范围
- 包含什么
- 明确不包含什么

## 设计
- 数据模型/迁移
- API 契约
- 页面与交互
- 权限与安全
- 失败与回滚路径

## 验收标准
- Given / When / Then 场景
- 自动化测试
- 手工验收步骤

## 发布影响
- 环境变量
- 数据迁移
- 兼容性
- 回滚方案
```

执行约束：

- 一个 PR 只解决一个清晰目标，避免同时重构无关模块。
- 数据库迁移一经合并不得修改已成功执行的版本，修复必须新增迁移。
- 新写接口必须同时提交权限测试、异常测试和接口文档。
- 新页面必须包含加载、空数据、失败、无权限和移动端状态。
- 任何涉及订单、库存、支付的改动必须说明幂等和回滚路径。
- PR 合并前必须通过 CI；部署后执行冒烟测试并保留版本与迁移记录。

## 12. 最终验收场景

`1.0.0` 发布前必须完整演示并留存测试结果：

1. 在空服务器安装系统并创建唯一店主。
2. 店主设置店名、Logo、联系方式和配送说明。
3. 店主创建一个包含多个 SKU 和图片的商品，录入库存并上架。
4. 消费者在桌面或手机端找到商品，通过 AI 获得基于真实数据的推荐。
5. 消费者注册、选择 SKU、加入购物车并使用线下付款或货到付款下单。
6. 商家在工作台查看订单、确认处理、填写承运商和运单号并发货。
7. 消费者查看物流信息并确认完成。
8. 取消或超时订单正确释放库存，重复操作不破坏状态。
9. 商品下架后消费者和 AI 均不再推荐或允许购买。
10. 执行备份，在另一套空环境恢复，并核对商品、订单、库存和媒体。

只有这一场景可稳定、可重复地完成，项目才从“可演示商城”进入“可供单个商家试点使用”的状态。
