# PhoneMall 项目改进计划

> 审查日期：2026-07-13
> 适用范围：后端微服务、Vue 前端、数据库初始化、Redis/RabbitMQ、AI 导购、Docker 与部署脚本
> 当前结论：项目具备毕业设计/功能演示所需的主要模块，但干净环境启动、订单库存一致性、访问控制和公开部署安全尚未达到可验收状态，更不应直接用于生产环境。

## 实施状态（2026-07-13）

本计划的 P0/P1 基线已在 `codex/project-hardening` 分支落实并由 CI 保护：

- 已移除仓库中的默认密钥与口令依赖，新增 `.env.example`，部署脚本会拒绝空密钥或不足 32 字节的 JWT 密钥。
- 已补齐初始化 SQL 字段、引入 Flyway 迁移，并将订单创建改为同步事务，移除未闭环的 RabbitMQ 核心下单路径。
- 已实现库存“预留 → 支付确认 / 取消释放”的状态机；数量、分页、地址、购物车和订单输入均有服务端校验。
- 已修复地址和购物车的跨用户访问，区分 access/refresh token 并在前端实现单飞刷新与退出登录。
- AI 对话要求登录、限制输入和网关 QPS，并使用有界 Spring 线程池；缺少密钥时可安全降级。
- 已收紧 CORS、服务网络和端口暴露，修复 Nginx 非法配置及部署脚本的失败传播。
- 已加入 Maven Wrapper、JWT 回归测试和 GitHub Actions；本地 Java 17 `./mvnw verify`、前端构建、依赖审计和两份 Compose 配置均已通过。

仍明确不在当前“毕业设计演示基线”范围内的生产能力：真实支付、退款/物流、管理后台与审计、数据备份恢复、跨地域高可用、AI 费用预算与内容安全审核。它们应作为生产化下一阶段工作，而不能在 README 或答辩材料中宣称已完成。

## 1. 改进目标

按以下三个交付等级推进，避免把“演示可用”和“生产可用”混为一谈：

1. **L1：可重复演示**：全新克隆后可按 README 一次构建、初始化并启动；注册、登录、商品、购物车、下单、支付/取消、AI 导购均可完整演示。
2. **L2：可公开测试**：不存在已知越权、伪造令牌、默认口令、调试后门和可轻易刷爆 AI 费用的问题；异常能被前端正确识别。
3. **L3：生产候选**：具备可靠消息、数据库迁移、自动化测试、监控告警、备份恢复、TLS 和发布回滚能力。

本轮优先完成 L1 和 L2。L3 中的真实支付、物流、后台运营等能力需要根据项目定位另行决定。

## 2. 已确认问题清单

### 2.1 P0：必须先修复

| 编号 | 问题 | 影响 | 主要证据 | 验收标准 |
| --- | --- | --- | --- | --- |
| P0-01 | JWT、MySQL、RabbitMQ 使用仓库内固定密钥/口令 | 已知 JWT 密钥可用于伪造任意用户；基础设施可能被直接接管 | `gateway-service/.../application.yml`、`user-service/.../application.yml`、`order-service/.../application.yml`、`docker/*.yml` | 仓库不再包含真实默认密钥；缺少必填变量时服务拒绝启动；旧密钥全部轮换 |
| P0-02 | 初始化 SQL 缺少实体已使用的四个字段 | 标准 `docker-compose.infrastructure.yml` 新装后，商品/购物车查询可能因未知列失败；只有 E5 脚本临时执行 ALTER | `Product.colorOptions/storageOptions`、`ShoppingCart.selectedColor/selectedStorage`，但 `sql/init.sql` 无对应列 | 在空数据库上执行正式迁移后，四个服务能启动并完成冒烟测试；不依赖部署脚本临时 ALTER |
| P0-03 | 下单、购物车和测试下单未校验 `quantity > 0` | 负数数量会增加 Redis 可用库存、产生负金额订单；空值会造成 500 | `OrderSubmitDTO` 无约束，多个 Controller 使用原始 `Map` | 0、负数、空值、超大值均返回明确 400；Redis/MySQL/订单数据完全不变化 |
| P0-04 | 库存状态机和 MySQL/Redis 语义不一致 | 正常购买 1 件即可形成 `total=99, locked=-1, available=100`；取消/超时不能正确恢复 MySQL，重启后重新污染 Redis | `StockServiceImpl.confirmDeduct()` 与 `restore()` | 任意订单状态下满足已定义库存不变量；支付、取消、超时、消费重试和服务重启后库存一致 |

推荐库存不变量：待支付时 `total = available + locked`；支付成功时同时减少 `total` 和 `locked`；取消/超时时减少 `locked` 并增加 `available`。如果保留 Redis Lua 预扣，必须增加可追踪的库存预留记录与补偿/对账任务，不能只依靠三个裸 Hash 字段。

### 2.2 P1：公开测试前必须修复

| 编号 | 问题 | 影响 | 主要证据 | 验收标准 |
| --- | --- | --- | --- | --- |
| P1-01 | 地址详情缺少用户归属校验 | 任意登录用户可按 ID 读取他人姓名、电话和地址 | `AddressController.getById()`、`AddressServiceImpl.getById()` | 用户 A 请求用户 B 地址时返回 404/403，且响应与日志不泄露地址存在性 |
| P1-02 | 购物车规格更新缺少用户归属校验 | 登录用户可修改他人的购物车规格 | `CartController.updateOptions()` 未传 userId；`ShoppingCartServiceImpl.updateOptions()` 仅按 cartId 更新 | 所有购物车写操作都以 `userId + cartId` 为条件，并有跨用户集成测试 |
| P1-03 | 业务异常 HTTP 状态与响应体协议不一致 | `BizException` 返回 HTTP 200，Axios 将库存不足、重复下单等当作成功 | `GlobalExceptionHandler.handleBizException()` 与前端 `request.ts` | 统一采用“正确 HTTP 状态 + 业务响应体”；前端对非成功 HTTP 或 `code != 200` 都进入统一错误分支 |
| P1-04 | Access Token 与 Refresh Token 无类型区别 | Refresh Token 可直接访问所有受保护 API；禁用用户仍可刷新；前端 401 后直接登出 | `JwtUtil.generateToken()`、`UserServiceImpl.refreshToken()`、前端 `request.ts` | Token 含 `type`/`jti`；网关只允许 access 访问业务 API；刷新接口只接受 refresh；刷新轮换并复查用户状态 |
| P1-05 | 调试/管理接口对普通用户开放 | 普通用户可创建“已支付测试订单”或触发全量 AI 商品改写和付费调用 | `/api/order/test-create`、`/api/order/ai/enrich` | 生产构建移除测试接口；管理接口需要管理员角色、审计日志和防重复任务机制 |
| P1-06 | AI 聊天匿名开放且资源无界 | 可被刷 API 费用、数据库记录和 JVM 线程；每个请求直接 `new Thread()` | 网关白名单 `/api/order/ai/chat`、`AiServiceImpl.chat()` | 采用登录/匿名额度、按用户/IP 限流、最大输入长度和有界线程池；达到限额返回 429 |
| P1-07 | 生产前端默认请求 `http://localhost:8080` | 部署后浏览器会访问访客自己的 localhost，前后端联调失败 | `request.ts`、`order.ts`、`AiAssistant.vue`；部署脚本未设置 `VITE_API_BASE_URL` | 生产默认使用同源相对路径 `/api`；开发环境通过 Vite proxy 或显式环境变量访问 8080 |
| P1-08 | 标准启动脚本首次启动会被 `set -e` 中断 | `docker rm -f` 删除不存在的容器时返回非零，首次 `start` 直接退出；构建还硬编码用户 `jill` | `start.sh` 第 65、81、103 行附近 | 新机器连续执行 start/stop/restart 均幂等；使用当前用户；脚本错误能明确退出并说明原因 |
| P1-09 | E5 发布脚本吞掉 Maven 构建失败 | 旧 JAR 存在时可能在编译失败后继续部署旧版本 | `e5-deploy.sh` 的 `mvn ... | grep ... || true` 与仅检查文件存在 | 启用 `set -o pipefail`；构建失败立即停止；使用 `clean package`，并校验产物时间/哈希 |
| P1-10 | 应用容器无重启策略和健康检查 | JVM 崩溃或服务器重启后应用不会自动恢复，脚本仍可能报告部署完成 | `docker run --rm -d` 未设置 `--restart`；健康检查失败仅 warn | 所有应用容器具备 restart policy、健康检查；关键服务不健康时部署失败 |
| P1-11 | 中间件和业务端口直接暴露 | MySQL、Redis、RabbitMQ、Nacos 及 host-network 服务可被局域网/公网直接访问 | Compose 的 `3306:3306` 等映射与业务服务 `--network host` | 非必要端口仅绑定 `127.0.0.1` 或内部 Docker 网络；Redis/Nacos/RabbitMQ 启用认证；外部只开放 80/443 |
| P1-12 | 订单异步链路没有完整可靠性闭环 | 发布成功不代表落库成功；消息丢失/毒消息会造成库存永久锁定；取消队列无消费者 | `RabbitTemplate` 无 confirms/returns；`order.cancel.queue` 只有声明无监听器 | 发布确认、消费幂等、有限重试、DLQ 处理、补偿与对账都有自动化用例和可观测指标 |

## 3. 推荐实施顺序

### 阶段 A：建立可复现基线

- 增加 Maven Wrapper，并固定 Java 17、Maven、Node 版本。
- 引入 Flyway，把 `init.sql` 转为首个正式迁移，补齐四个缺失字段。
- 新增 `application-local.yml`；所有密码、JWT、AI Key 改为环境变量，生产环境禁止默认值。
- 提供 `.env.example`，仅包含变量名和安全说明。
- 修复 `start.sh`、E5 构建失败传播、无效 Nginx 配置。

验收：在没有历史数据库、target、node_modules 和旧容器的环境中，仅按 README 可以完整启动；重复执行启动/停止不失败。

### 阶段 B：修复订单与库存

- 为购物车、下单、地址、分页和 AI 查询定义专用 DTO，使用 Bean Validation；禁止 Controller 接收原始 `Map` 或持久化实体作为写入 DTO。
- 明确库存状态机，重写预扣、确认、取消、超时逻辑。
- 在数据库层增加库存和数量约束；任何异常不得产生负库存、负锁定或负金额。
- 调整幂等键生命周期；保存订单价格快照。
- 决定异步下单实现：
  - **推荐的最小可靠方案**：同步创建可查询的 processing 订单和 outbox 事件，事务提交后发布 MQ；消费者幂等推进状态。
  - 若毕业设计无需展示可靠消息，可改为同步落库，把 MQ 仅用于非核心通知，优先保证正确性。
- 增加 publisher confirms/returns、有限重试、DLQ 消费和库存对账任务。

验收：并发 1000 次、库存 100 时恰好 100 次成功；支付、取消、超时、重复消费、MQ 暂停与服务重启后库存不变量仍成立。

### 阶段 C：修复认证、授权与隐私

- 轮换全部已提交凭据；内部服务和中间件改为私有网络。
- Token 增加类型、ID、签发方和受众；刷新时校验用户状态并轮换 refresh token。
- 在网关或统一安全组件中集中处理身份；下游接口仍需执行资源归属校验。
- 修复地址和购物车 IDOR；移除/隔离测试下单；AI enrich 仅管理员可用。
- 登录与 AI 增加按用户/IP 的限流；限制 AI 输入和并发。
- 删除 PII 日志，定义 AI 对话保存与删除策略。

验收：跨用户访问测试全部失败；refresh token 不能调用业务 API；禁用用户无法刷新；无认证用户不能触发付费 AI（除非明确配置匿名额度）。

### 阶段 D：统一前后端协议与部署

- 统一 HTTP 状态与 `R.code`，前端响应拦截器统一判断业务成功。
- 默认使用同源 `/api`，开发环境使用 Vite proxy；AI 请求复用同一 API 配置。
- 前端实现单飞刷新令牌、请求重放和刷新失败登出。
- 异步订单提供 processing 页面和重试/轮询，不提前删除购物车。
- 应用容器增加重启策略、健康检查、资源限制与结构化日志。
- 公网入口启用 TLS，收紧 CORS 和端口暴露。

验收：生产构建在另一台电脑访问时不会请求 localhost；所有业务失败都显示后端真实消息；断开/恢复任一依赖后服务状态可观察且能恢复。

### 阶段 E：AI 与检索工程化

- 缺少 API Key 时禁用 AI Bean/路由并提供清晰能力状态，不在启动线程中盲目请求外部 API。
- 使用受 Spring 管理的有界执行器；客户端断开时取消外部请求。
- 修复重复历史消息，限制历史长度/字符数，显式使用 UTF-8。
- 将 enrich 改为管理员异步任务，支持进度、幂等和失败重试。
- 若商品规模增长，再将内存向量升级为持久化向量库；当前规模可继续内存实现，但必须支持商品变更后的增量更新和缓存失效。

验收：并发达到配置上限时返回 429 而不是持续创建线程；缺 Key 时电商功能正常且 AI 页面显示“未配置”；AI 调用有耗时、失败率和费用指标。

## 4. 自动化测试矩阵

### 后端单元/集成测试

- 注册、登录、禁用用户、access/refresh 类型与过期。
- 地址和购物车跨用户读取/写入。
- 数量为 null、0、负数、极大值；缺少收货字段。
- 商品不存在、下架、库存不足、重复下单。
- 正常预扣、支付确认、主动取消、超时取消。
- MQ 重复消息、乱序、消费异常、DLQ、发布失败与补偿。
- Redis 清空、服务重启后从 MySQL 恢复并保持库存不变量。
- 商品分页/热门/AI 查询的边界值。

建议使用 JUnit 5、Spring Boot Test、Testcontainers（MySQL/Redis/RabbitMQ）和 MockWebServer/WireMock（AI API）。

### 前端测试

- 响应体 `code != 200` 与 HTTP 4xx/5xx 均显示正确错误。
- access 过期后仅刷新一次，并重放并发请求。
- 生产 API 地址为同源；AI SSE 能解析分片并正确结束/报错。
- 下单 processing、成功、失败与购物车保留/删除行为。
- 路由鉴权和跨页面登录状态恢复。

### 端到端测试

- 新用户注册 → 登录 → 浏览商品 → 加购物车 → 选择地址 → 下单 → 支付。
- 新用户注册 → 下单 → 取消；下单 → 自动超时。
- 两用户互相访问地址、购物车、订单资源均失败。
- AI 未配置、配置成功、限流、供应商超时四种场景。
- 全新数据库迁移和从上一版本升级两种场景。

## 5. CI 与发布门禁

每次提交至少执行：

```bash
./mvnw test
npm ci --prefix biyesheji-frontend
npm run build --prefix biyesheji-frontend
npm audit --prefix biyesheji-frontend --audit-level=high
docker compose -f docker/docker-compose.infrastructure.yml config -q
```

合并/发布前增加：

- Testcontainers 集成测试与端到端冒烟测试。
- 空数据库迁移和升级迁移测试。
- 镜像/依赖漏洞扫描。
